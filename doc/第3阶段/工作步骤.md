## 核心建模（技术大纲 §3.3 / §4.1.2）

外生地面 OD 需求的合成遵循：

\[
D_g(t) = w_g \cdot [1 + a\cos(2\pi(t - t_{\text{noon,local}}(g))/24\text{h})] \cdot S(t) \cdot k_0
\]

- \(w_g\)：WorldPop 区域需求权重（P1 已算，和为 1），决定**空间分配**；
- \(a\)：昼夜调制幅度（config `demand.diurnal_amplitude_a`，0.4）；
- \(t_{\text{noon,local}}(g)\)：像元 g 的当地正午 UTC 时刻 = 12 − `utc_offset(g)`；
- \(S(t)\)：GÉANT 清洗后全局总强度（P1 已算），决定**时间变化**；
- \(k_0\)：总量放缩因子，P3 标定。

**关键**：地面需求是"区域→区域"的 OD，但本研究的路由对象是"卫星→卫星"的 commodity。P3 把每个地面像元映射到其主服务卫星（P2 覆盖），于是：
- **源**：像元 g 的需求 → 接入卫星 `src_sat(g,t)`；
- **目的**：像元 g 的需求要去往的目的地——这里需要一个"目的区域"模型。

**目的区域模型（P3 决策）**：真实地面流量是任意区域对。本研究简化为：每个像元 g 产生需求，目的为**按人口权重随机抽取的另一个像元 h**（h 的接入卫星为目的卫星）。这保证目的卫星分布与人口分布一致，且 OD 是动态的（每时隙或固定模式）。具体见步骤 3。

## 步骤 2：基础地面需求（时间 × 空间 × 昼夜）（WRITE）

### 2.1（WRITE）`src/data/demand.py`

实现基础地面区域需求（技术大纲 §4.2 步骤 4）：

- `compute_base_demand(S_clean, times, worldpop_weights, cfg) -> dict`：
  - 对每个时隙 t、每个有效像元 g：
    - 昼夜调制：`D_g(t) = w_g * (1 + a*cos(2π*(t_hours - local_noon_g)/24)) * S(t) * k0`
    - `local_noon_g = 12 - utc_offset(g)`（小时）
    - `t_hours` = 时刻的小时浮点数
  - **但逐像元逐时隙计算 5124万×56966 不可行**。利用 WorldPop 权重的结构优化：
    - 权重 w_g 不随时间变，只有昼夜调制随时间；
    - 按 UTC 偏移**分桶**（offset 以 15min 为粒度量化到 ~96 桶，实际经度/15 后量化），每桶内权重求和 `W_b = Σ_{g∈bucket b} w_g`；
    - 每时隙每桶算一个昼夜系数，桶级需求 = `W_b * diurnal_b(t) * S(t) * k0`；
    - 像元级需求按桶内 w_g 比例分配（P3 不需逐像元，只需桶级 + 接入卫星聚合，见步骤 3）。
  - 返回桶级时间序列 + 桶→像元映射。

### 2.2（VERIFY）

```bash
python -c "
import sys; sys.path.insert(0,'.')
from src.common import load_config, resolve_paths
from src.data import io
from src.data.demand import compute_base_demand
cfg = load_config(); p = resolve_paths(cfg)['data_interim']
S = io.load_npz(p/'geant'/'S_clean.npz')['S_clean']

wp = io.load_npz(p/'worldpop'/'weights.npz')

print('桶数:', bd['n_buckets'], '| 桶权重的和:', bd['bucket_weights'].sum())
print('t=0 总需求:', bd['bucket_demand_ts'][0].sum(), '| S(0):', S[0])
"
```

期望：桶权重的和≈1.0；t=0 总需求≈S(0)×k0（昼夜调制在 t=0 全局平均≈1）。

## 步骤 3：混合 OD 配对与 commodity 聚合（WRITE）

### 3.1（WRITE）`src/data/od_pairs.py`

实现地面 OD 配对 → 卫星 commodity（技术大纲 §4.2 步骤 6）。采用**随机+重力+热点混合模型**（P3 决策，用户确认）：

每个源桶 b_s 的需求按比例分成三股流，各自选目的桶 b_d：

```
π_rand · 随机股：目的按桶权重独立抽样 → 全局扩散、背景流量
π_grav · 重力股：D ∝ W_{b_d} / dist(b_s,b_d)^γ → 空间相关性、近邻走廊
π_hot  · 热点股：预定义热点对 → 显式高带宽走廊、可控压力源
(π_rand + π_grav + π_hot = 1，默认 0.4/0.4/0.2)
```

- **随机股**：`np.random.choice(buckets, p=normalized_W)` 按桶权重抽目的，用 `make_seed_stream(cfg.seed.data,'od')` 派生独立 Generator（可复现，非全局 np.random）；
- **重力股**：源 b_s 到各 b_d 的分配概率 `P(d|s) ∝ W_d / dist(s,d)^γ`，γ=1.5，dist 为**地面大圆距离**（源/目的桶代表像元的经纬度）；预计算每对的归一化分配矩阵（桶数 ~200，矩阵 200×200 可预存）；
- **热点股**：预定义 10 对热点桶（自动选，见 3.2），每对在双向分配 `π_hot/10` 的需求权重；热点对清单写入文档供审。
- **流量类型标记**：每条 OD 标 `flow_type ∈ {random,gravity,hotspot}`，供 P4/P6 审计。
- **接入映射**：每个桶的代表像元经 `assign_primary_sat(positions[t], lon, lat, cfg)` 得 src_sat/dst_sat；覆盖变化慢，**每 10 时隙重算一次覆盖映射**，中间沿用。
- `build_od_commodity(base_demand, positions, times, worldpop_weights, cfg) -> dict`：
  - 对每时隙：三股分配 → 源/目的卫星 → 按目的卫星聚合 commodity；
  - 输出稀疏 commodity 表（每时隙 `(src_sat,dst_sat,volume,flow_type)`）；
  - 处理**无覆盖桶**（高纬，~4% 桶）：需求标记 `unserved`，不计入网络 commodity（P3 丢弃，占人口少）。

### 3.2（WRITE）热点对自动选择 `src/data/hotspots.py`

- `select_hotspots(worldpop_weights, K=10) -> list[(src_bucket, dst_bucket, label)]`：
  - 按 WorldPop 高人口密度区域（桶权重 top-N）选源/目的候选；
  - 覆盖主要人口走廊：东亚内部、欧洲内部、北美内部、跨洲各若干对；
  - 返回 10 对 + 人类可读标签（如"东亚-欧洲"），写入文档。

## 步骤 5：总量标定 k0（WRITE + RUN）★ 核心步骤 ★

### 5.1（WRITE）`src/data/calibrate.py`

实现总量↔容量标定（技术大纲 §4.1.4，唯一人工标定参数）：

- 标定目标：调 k0 使**链路利用率中位 ≈ 0.5、P95 ≈ 0.8**，高峰链路可触拥塞阈值 0.8。
- **难点**：链路利用率 = 链路承载流量 / 链路容量，而链路承载流量由 P4 路由策略决定，P3 还没有路由结果。
- **P3 的近似标定方法**：
  - 用一个**代理路由**（最短路径 Dijkstra）在 baseline 场景需求上跑一遍，得到各链路承载流量（P3 可调用 P4 的 Dijkstra——但 P4 还没实现）；
  - **替代**：P3 用一个轻量"全最短路径"流量分配估计链路负载（不实现完整仿真，只做最短路分流），算利用率分布，二分搜索 k0。
  - 或者：**P3 先用启发式估 k0**（基于需求总量 / 链路总容量 × 经验系数），P4 实现后再用真实路由 refine。
- **P3 决策**：采用启发式估计 + 在 P4 实现后回校。具体：
  - `estimate_k0(commodity_ts, topology, cfg) -> float`：
    - 估算全网需求总量 `D_total = Σ commodity`；
    - 估算全网链路总容量 `C_total = n_edges * link_capacity`；
    - 取 `k0_init = 0.5 * C_total / D_total`（使平均利用率≈0.5），再按经验上调使 P95≈0.8；
    - 用最短路分流（networkx all_pairs_shortest_path + 流量按路径分配）算各链路负载分布，二分 k0 到 P95≈0.8。
