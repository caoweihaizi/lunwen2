## 2. 数据产出（`data/processed/sim/`，被 `.gitignore` 忽略）

```
data/processed/sim/
├── <policy>_<scenario>/        # 9 个 run
│   ├── link_state_shard0.npz   # 114 个分片，每 500 时隙一个
│   ├── link_state_shard1.npz
│   ├── ...
│   ├── link_state_shard113.npz
│   └── summary.json            # 该 run 汇总指标
```

每 shard：`link_state (N,10)` 列 = `[t, i, j, offered, carried, queue, drop, util, delay, dist]`，单位 Mb（流量×Δt）。

**流式落盘**：每 500 时隙 flush 一次，内存恒定 ~5GB（旧版累积 7255 万明细触发 52GB swap）。每 run 0.19GB，9 run 共 ~1.7GB。

**明细表**（node_commodity/link_commodity）默认不生成（`keep_detail=False`），审计时用单独脚本抽样启用。

## 3. 仿真特性（真实执行结果）

### 3.1 k0 refine（ECMP 真实路由）

| 项 | 值 |
|---|---|
| P3 k0（最短路代理） | 1.3554 |
| P4 factor | 2.9964 |
| **P4 k0（ECMP refine）** | **4.0612**（回填 config） |
| 标定目标 | P95≈0.45（中负载） |
| 实测 P95 | 0.450（refine 采样）/ 0.373-0.403（全量） |

**P95 口径决策**：66 星稀疏拓扑下 OD 流量只走 34% 链路，中位利用率恒为 0，技术大纲"中位 0.5"不可达。改用 P95 口径（记入 8.2 局限性）。

### 3.2 9 个 run 结果

| run | util P95 | 丢包率 | tot_offered(Mb) |
|---|---|---|---|
| dijkstra_baseline | 0.373 | 12.6% | 1.71e11 |
| dijkstra_burst | 0.377 | 13.8% | 1.76e11 |
| dijkstra_compound | 0.398 | 14.9% | 1.74e11 |
| ecmp_baseline | 0.379 | 10.8% | 1.71e11 |
| ecmp_burst | 0.385 | 12.0% | 1.76e11 |
| ecmp_compound | 0.403 | 13.6% | 1.74e11 |
| queue_aware_baseline | 0.380 | 10.8% | 1.71e11 |
| queue_aware_burst | 0.385 | 12.1% | 1.76e11 |
| queue_aware_compound | 0.403 | 13.6% | 1.74e11 |

**关键观察：**
1. **三策略产生不同分布**：ECMP/Queue-aware 丢包（10.8%）低于 Dijkstra（12.6%）——均衡分流生效 ✓
2. **场景效应递增**：baseline→burst→compound 丢包率递增（10.8%→12.0%→13.6%），突发+故障加剧拥塞 ✓
3. **Queue-aware ≈ ECMP**：两者结果几乎相同（见 §4 已知问题）
4. **守恒精确**：每个 shard offered=carried+drop，差 0.0000 ✓

### 3.3 故障

compound 场景：11 条链路全程失效（10%），故障链路不承载数据（carried=0）。

### 3.4 队列参数（P4 决策）

- 缓存容量 B = 1 时隙容量 = C×Δt
- 最大等待 = 2 时隙（6min）
- 超时丢包 + 缓存溢出丢包

## 4. 已知问题（不阻断，记录处理）

| 问题 | 原因 | 处置 |
|---|---|---|
| e2e 时延=0.0ms | e2e_buf 在 flush 时被清空，最后取均值是空 | P11 精确重算 e2e（P4 辅助指标，不影响 link_state 真值） |
| Queue-aware ≈ ECMP | 66 星稀疏拓扑下多数 OD 的势值下降邻居唯一；α·φ 量级淹没 ρ/q | 记录为设计局限；P10 MAPPO（RL）会有更大差异；可在 P4 调参归一化代价 |
| shard 数 114/run | 每 500 时隙一文件 | 可合并，但不影响使用 |

## 5. 给 P5/P7/P9 的接口契约

1. **link_state**：`np.load('sim/<run>/link_state_shard{N}.npz')['link_state']` → (N,10)，P7 训练/P9 校准的真值来源；
2. **场景**：baseline/burst/compound，compound 含 11 条故障链路；
3. **k0**：已回填 config（4.0612），commodity 已应用；
4. **守恒**：每 shard offered=carried+drop，可用于 P6 审计；
5. **明细表**：默认未生成，审计需 `keep_detail=True` 重跑抽样时段。

**P4 不交付**：遥测缺失（P5）、模型训练（P7）、校准（P9）、RL 路由（P10）。
