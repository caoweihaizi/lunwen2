## 仿真器物理模型（技术大纲 §3.2 / §3.5）

### 流级逐跳路由（§3.5）

每个时隙 t：
1. **commodity 聚合**：所有源不同但目的卫星 d 相同的业务聚合为 commodity (·, d)；
2. **逐跳分配**：节点 i 为每个活跃 (i,d) 选一个下一跳 j（策略决定）；
3. **链路汇总**：各目的流分配结果汇总到输出链路 (i,j)；
4. **链路服务**：每条链路按容量 C_ij 服务队列，更新队列/利用率/时延/丢包。

### 队列离散更新（§3.2）

\[
Q_{ij,t+1}=\min\{B_{ij},\max(0,Q_{ij,t}+A_{ij,t}-S_{ij,t})\}
\]

- \(A_{ij,t}\)：时隙 t 到达链路 (i,j) 的业务量（commodity 分配汇总）；
- \(S_{ij,t}\)：成功发送量 = min(队列+到达, C_ij×Δt)；
- 超过 B 或等待超 max_wait 的丢包。

### P4 敲定参数（用户确认）

| 参数 | 值 | 说明 |
|---|---|---|
| 链路容量 C_ij | 10 Gbps（config） | 每时隙可服务 C×Δt = 10Gbps×3min = 1800 Gb |
| 缓存容量 B_ij | **1 时隙容量** = C_ij×Δt | 队列上限=1时隙服务量，超则丢包 |
| 最大等待 | **2 时隙**（6min） | 超时丢包 |
| 故障率 | 10% | 随机关闭链路（运行拓扑） |

## 步骤 1：仿真器核心（WRITE）★ 工程重心 ★

### 1.1（WRITE）`src/sim/network.py` — 网络状态

`class LinkState`：每条有向链路保存 capacity、distance、prop_delay、queue、utilization、drop_count、offered_load、carried_load。

`class NetworkState`：
- 从 P2 稀疏边表重建某时隙邻接 + 链路属性；
- 持有所有有向链路的 LinkState；
- `step(commodity, routing_decisions, cfg)`：执行一个时隙的链路服务与队列更新；
- 故障链路：从邻接移除，不承载数据。

### 1.2（WRITE）`src/sim/queue.py` — 队列更新

实现 §3.2 离散队列更新：
- `update_queue(link, arrival, cfg) -> (served, dropped, queue_next)`：
  - `offered = link.queue + arrival`；
  - `served = min(offered, capacity_timeslot)`（capacity_timeslot = C×Δt）；
  - 超过 B 的部分立即丢弃（缓存溢出）；
  - 等待超 max_wait 的丢弃（超时，需记录入队时间）；
  - 返回 served/dropped/新队列。

**单位约定**：commodity volume 是 Mbps（速率）。时隙内流量 = volume × Δt（Mb）。队列单位用 Mb。容量 ×Δt = C(Gbps)×1000×Δt(s)/8 ... 统一用 Mb 或 Gb，**P4 统一用 Mb**：volume_mbps × (Δt_min×60) = Mb。

### 1.3（WRITE）`src/sim/simulator.py` — 仿真主循环

`class FlowLevelSimulator`：
- 输入：commodity 时间序列、计划拓扑边表序列、故障注入、路由策略；
- 每时隙：
  1. 重建该时隙运行拓扑（计划拓扑 − 故障链路）；
  2. 路由策略为每个活跃 (i,d) 决定下一跳；
  3. 汇总 commodity 到输出链路 → arrival；
  4. 链路服务 + 队列更新 → served/dropped/queue；
  5. 记录链路状态真值（offered/carried/queue/util/delay/drop）；
  6. 端到端时延：累加路径各跳 prop_delay + queue_delay。
- 输出：四张逻辑表（见步骤 4）。

### 1.4（WRITE）`tests/test_simulator.py`

断言：
- 业务守恒：Σ served + Σ dropped + Σ queue_next == Σ offered（每时隙）；
- 故障链路不承载数据（carried=0）；
- 队列不超 B；
- 端到端时延 ≥ 路径传播时延；
- 无环：路径不重复访问节点（用势函数保证，见步骤 2）。

## 步骤 2：三类离线策略（WRITE）

### 2.1（WRITE）`src/sim/policies.py`

实现三类离线行为策略（技术大纲 §4.2 步骤 8）：

**Dijkstra（最短传播时延）**：
- 每时隙对每个目的 d，用传播时延作权重算全源最短路；
- 每个节点 i 的下一跳 = 最短路上 i 的后继；
- 势函数 φ = 最短时延。

**ECMP（等价多路径）**：
- 对每个目的 d，找所有等价最短路径；
- 流量在分叉点均分到等价下一跳；
- split_ratio 记录分流比例；
- 势函数 φ = 最短跳数。

**Queue-aware stochastic routing**（技术大纲 §4.2 公式）：
- 局部代价 \(c_{ij,d,t}=\alpha\phi_t(j,d)+\beta\rho_{ij,t}+\gamma q_{ij,t}+\eta\delta_{ij,t}\)；
  - φ：目的势值；ρ：利用率；q：队列；δ：传播时延；
- 按温度 T 软max采样：\(P(j|i,d,t)=\exp(-c/T)/\sum\exp(-c/T)\)；
- 候选仅合法邻居（势值下降 + 可用）；
- **关键**：此策略需要当前队列/利用率状态（与仿真器交互），是 stateful 的；Dijkstra/ECMP 是 stateless（仅拓扑）。

### 2.2（WRITE）`src/sim/potential.py`（复用 P2）

P4 的势函数用 P2 `potential_shortest_hops` / `potential_propagation_delay`，但在**运行拓扑**（故障后）上重算（技术大纲 §6.3）。

### 2.3（WRITE）`tests/test_policies.py`

断言：
- Dijkstra：下一跳在最短路上，势值严格下降；
- ECMP：分流比例和=1，等价路径均分；
- Queue-aware：采样概率合法（和=1），高负载链路被选概率低；
- 无合法邻居时执行等待动作。

## 步骤 3：故障注入（WRITE）

### 3.1（WRITE）`src/sim/failure.py`

`inject_failures(edge_lists, cfg, rng, scenario) -> failure_log`：
- 按 `link_failure_rate=0.10` 随机关闭链路；
- **故障先于路由生效**（技术大纲 §4.5.2）：在每时隙（或固定时段）的计划拓扑上移除故障链路 → 运行拓扑；
- 故障链路状态：`link_active=False, failure_flag=True`，立即可知（§4.5.2）；
- 场景：baseline 无故障；compound = burst demand + 10% 故障；
- 故障可**固定**（整个仿真期同一组链路失效）或**时变**（每段重新注入）；P4 采用**固定故障集**（一组链路全程失效），更可控可复现。

### 3.2（VERIFY）

确认故障链路不在运行拓扑邻接中；故障率实际=10%。

## 步骤 5：ECMP refine k0（WRITE + RUN）

### 5.1（WRITE）`src/sim/refine_k0.py`

用 ECMP（均衡分流）在 baseline 上跑，重算利用率分布：
- 若中位 <<0.3，说明 k0 偏小（流量集中在少数链路）；若中位>>0.5，k0 偏大；
- 目标：中位∈[0.3,0.5]、P95∈[0.7,0.9]、max≤1.5；
- 二分 refine k0，更新 config `demand.k0`，重生成 P3 commodity（调用 P3 的 scale）。

### 5.2（注意）

refine k0 会改变 commodity，P4 所有策略重跑。P4 流程：先 refine → 再用最终 k0 跑全部策略×场景。

## 执行注意事项（给 Claude Code）

1. **复用 P0–P3 接口**，不另造。
2. **顺序执行**：网络状态→队列→仿真器→策略→故障→逻辑表→refine k0→合并→入口。
3. **不越界**：P4 不做缺失（P5）、模型（P7+）、校准（P9）、RL 路由（P10）。
4. **正确性优先于速度**：仿真器是 P6 硬门控的基础，守恒/无环/故障检查必须严格；先小规模（前 1000 时隙）验证再全量。
5. **单位统一**：全程 Mb（流量×Δt），避免 Mbps/Gbps 混淆。
6. **可复现**：故障注入、Queue-aware 采样用 `make_seed_stream` 派生独立 Generator。
7. **k0 refine 在前**：先 refine 再跑全部 run，避免用错 k0 重跑。
8. **耗时管理**：9 run 全量可能很久；若超时，先跑 baseline×3策略 + 采样时段验证正确性，全量留待确认后跑。
9. **VERIFY 必跑**，尤其守恒与无环。
10. **完成标志**：步骤8 全✓ + `run.sh P4 0` 成功 + 产出/审计文档 + git commit。
